# RLSV Autopost — доработка фронтенда, аудит и рекомендации

Документ описывает изменения, внесённые в модуль `rlsv_autopost` (CS‑Cart 4.9.2 / PHP 7.x), порядок установки/миграции, а также аудит кода, логики и БД с предложениями по улучшению — включая публикацию через API внешних площадок с учётом их ограничений.

> Важно про цифры лимитов внешних API: значения квот/ограничений ниже приведены как ориентир «эпохи модуля» и **меняются провайдерами**. Перед боевым запуском сверяйте актуальные значения в dev‑консолях площадок. Код намеренно построен так, чтобы корректно реагировать на лимиты (бэк‑офф, повторы, журналирование), а не «зашивать» конкретные числа.

## 1. Что добавлено (клиентская часть / фронтенд)

Вся новая функциональность доступна авторизованному пользователю витрины. Пользователь видит и меняет **только свои** данные; товары — только те, что назначены ему администратором.

1. **Страница настроек API и прокси** — `index.php?dispatch=rlsv_account.settings`
   Пользователь заполняет токены площадок (VK, Instagram, Telegram, OK, Pinterest, MAX, YouTube, RuTube) и адрес **прокси‑сервера**, с которого публикуется контент. Сохранение защищено CSRF‑токеном, принимаются только поля из белого списка, прокси проходит валидацию формата, токены пишутся **в зашифрованном виде** (см. §3).

2. **Аналитика публикаций с вкладками** — `index.php?dispatch=rlsv_account.stats`
   - вкладка **«Обзор»** — сводные счётчики (всего/опубликовано/ошибки/в очереди), разбивка по площадкам, журнал последних публикаций;
   - **по одной вкладке на каждый канал** (площадку) — счётчики и журнал по конкретному каналу;
   - вкладка **«По товарам»** — агрегаты по каждому товару, по которому шла публикация (всего/успешно/ошибки, дата последней публикации, разбивка по площадкам, ссылка на редактор контента).

3. **Редактирование «подробного описания» товара на витрине** с настройками безопасности:
   - встроенный редактор прямо на карточке товара (хук `products:description`) — виден только пользователю, которому товар назначен;
   - отдельная страница редактора — `index.php?dispatch=rlsv_account.product&product_id=ID`;
   - сохранение защищено CSRF, проверкой прав на товар и **серверным санитайзером HTML** (allowlist тегов/атрибутов, удаление `script/style/iframe`, `on*`‑обработчиков и `javascript:`/`data:` URI);
   - там же можно включить/выключить автопостинг товара и задать URL видеопрезентации.

4. **Поле прокси‑сервера для профиля** — присутствует и в кабинете витрины, и в карточке администратора. Значение применяется ко всем исходящим запросам публикации этого профиля (ВК/IG/TG/OK/Pinterest/MAX/YouTube/RuTube и скачивание медиа).

5. **Панель «Автопостинг» в кабинете покупателя** (хук `profiles:account_information`) — быстрые ссылки на настройки и аналитику.

## 2. Установка и миграция

1. **Залейте файлы** модуля поверх существующих (структура папок сохранена).
2. **Импортируйте языковые переменные**: Администрирование → Языки → импорт `var/langs/ru/...po` и `var/langs/en/...po` (либо переустановите модуль на dev‑копии).
3. **Задайте ключ шифрования токенов**: Настройки модуля → «Ключ шифрования токенов» (`token_crypt_key`) — длинная случайная строка. Если оставить пустым, ключ выводится из install‑специфичных значений (хуже для переносимости, но рабочее). **Старые незашифрованные токены продолжат читаться** — перешифровка происходит при следующем сохранении профиля.
4. **Назначьте издателям товары/категории** (карточка администратора → вкладка «Автопостинг» → привязки). Это и есть «выдача прав» пользователю витрины на редактирование контента и постановку в очередь.
5. **Крон** оставьте как был: `index.php?dispatch=rlsv_oauth.cron&cron_password=...` (см. §3 о безопасности крона).
6. **Хуки тем**: имена хуков `products:description` и `profiles:account_information` присутствуют в стандартной теме `responsive`. Если в вашей кастомной теме они называются иначе и блоки не появляются — используйте прямые адреса (`rlsv_account.product`, `rlsv_account.settings`, `rlsv_account.stats`) и/или переименуйте файлы‑хуки под вашу тему.

### Миграция БД (для уже установленных копий)

В `addon.xml` движок таблиц изменён на **InnoDB**, но `CREATE TABLE IF NOT EXISTS` не пересоздаёт уже существующие таблицы. Для перевода действующей установки выполните вручную (с резервной копией!):

```sql
ALTER TABLE cscart_rlsv_autopost_queue      ENGINE=InnoDB;
ALTER TABLE cscart_rlsv_autopost_tokens     ENGINE=InnoDB;
ALTER TABLE cscart_rlsv_autopost_products   ENGINE=InnoDB;
ALTER TABLE cscart_rlsv_autopost_categories ENGINE=InnoDB;
ALTER TABLE cscart_rlsv_autopost_user_links ENGINE=InnoDB;
```
(Префикс `cscart_` замените на свой.)

## 3. Аудит безопасности

| # | Находка | Риск | Статус |
|---|---------|------|--------|
| S1 | **Токены площадок хранились в открытом виде** (JSON в `tokens`) | Компрометация БД/бэкапа = утечка всех токенов | **Исправлено**: AES‑256‑CBC (`openssl`), формат `enc:v1:`, ключ из `token_crypt_key`/ядра/derived. Обратная совместимость с легаси‑записями. Деградация до JSON только если `openssl`/ключ недоступны. |
| S2 | **Отсутствовала CSRF‑защита** форм | Подделка запросов на сохранение | **Исправлено** для всех новых форм витрины: сессионный nonce + `hash_equals`. |
| S3 | **Stored‑XSS через «подробное описание»** на витрине | Внедрение скриптов в публичную карточку | **Исправлено**: серверный санитайзер на `DOMDocument` (allowlist тегов/атрибутов, вырезание `script/style/iframe/object/embed/form/svg`, `on*` и `javascript:`/`data:`/`vbscript:` URI; `rel="nofollow noopener noreferrer"` на ссылки). Ограничение 50 000 символов. |
| S4 | **Авторизация на редактирование товара** | Эскалация: правка чужих товаров | **Исправлено**: правка разрешена только при админ‑привязке (`user_links`) — прямой или через категорию. Самопривязка с витрины невозможна. |
| S5 | **Open redirect** после сохранения | Фишинг через `redirect_url` | **Предотвращено**: редирект только на «свои» страницы (`products.view`/`rlsv_account.product`), произвольный URL не принимается. |
| S6 | Вывод в backend‑шаблонах без `|escape` (`{$t.vk_access_token}` и т. п.) | Reflected‑XSS в админке (низкий: данные вводит сам админ) | **Рекомендация**: добавить `|escape` в backend‑шаблонах. Все **новые** витринные шаблоны уже экранируют вывод. |
| S7 | `cron_password` передаётся в URL (GET) | Утечка через логи веб‑сервера/прокси/Referer | **Рекомендация**: принимать пароль и из заголовка/POST; не логировать query‑string; рассмотреть запуск по cli. |
| S8 | **SSRF при скачивании медиа** (`fn_rlsv_autopost_download_to_tmp`) и через пользовательский **прокси** | Запросы к внутренним адресам | **Частично**: прокси валидируется по схеме/хосту/порту. **Рекомендация**: для `image_url`/`video_url` — резолвить хост и блокировать приватные диапазоны (10/8, 172.16/12, 192.168/16, 127/8, link‑local), запрет редиректов на приватные адреса, лимит размера. |
| S9 | `feed_secret` сравнивается обычным `!==` | Тайминг‑атака (низкий) | **Рекомендация**: `hash_equals`. |
| S10 | Размер скачиваемого видео/файла не ограничен | DoS/диск | **Рекомендация**: лимиты по размеру и таймауту; стрим в файл. |

### Замечания по реализации безопасности

- **Санитайзер** дизайн‑зависимостей не имеет, но для промышленного уровня рекомендуется **HTMLPurifier** (точнее обрабатывает крайние случаи: вложенные кавычки в атрибутах, экзотические сущности, CSS в `style`). В текущей версии `style` удаляется целиком — это безопасный, но строгий выбор.
- **Шифрование**: смена `token_crypt_key` делает ранее сохранённые токены нечитаемыми (декодер вернёт пустой массив, не роняя процесс). Это ожидаемое поведение; задокументировано.

## 4. Аудит логики и корректности

| # | Находка | Статус |
|---|---------|--------|
| L1 | **Гонка при обработке очереди**: `SELECT pending` → затем безусловный `UPDATE processing`. Два параллельных крона могли взять одну задачу. | **Исправлено**: атомарный захват `UPDATE ... SET status='processing' WHERE queue_id=? AND status='pending'` с проверкой числа затронутых строк; «не своя» задача пропускается. |
| L2 | **Повтор без паузы**: при ошибке задача возвращалась в `pending` с прежним `run_after` → мгновенный ретрай, давление на лимиты API. | **Исправлено**: экспоненциальный бэк‑офф `run_after = now + 60·2^attempt` (2/4/8 мин). |
| L3 | **`REPLACE INTO` для настроек товара/категории/токенов**: REPLACE = DELETE+INSERT, сбрасывает несвязанные поля и «дёргает» AUTO_INCREMENT/триггеры. | **Рекомендация**: использовать `INSERT ... ON DUPLICATE KEY UPDATE`. Для текущих таблиц некритично (все поля переписываются), но REPLACE опаснее при расширении схемы. |
| L4 | **Легаси‑фолбэк** (`get_target_users`) берёт всех админов целевой группы, если нет явных привязок. | Корректно для админ‑сценария; но при переходе на витринных издателей **полагайтесь на явные привязки** (фолбэк ограничен `user_type='A'`, поэтому витринных пользователей не затронет случайно). |
| L5 | Снимок карточки (`payload`) фиксируется в момент сохранения товара. Если описание поправили на витрине **после** постановки в очередь — уйдёт старый текст. | **Поведение by design**. При необходимости — пересобирать карточку в момент отправки кроном (компромисс: «что сохранил — то и ушло» против «самое свежее»). |
| L6 | OK: загрузка фото **не реализована** (только текстовый media‑topic). | Документировано в коде. Требует `photosV2.getUploadUrl` → upload → `photos.commit` (см. §6). |
| L7 | Прокси хранится как обычный токен‑ключ `proxy`; применяется глобально на время диспетчеризации (`static`). | Работает, но при будущем параллелизме в одном процессе статик‑состояние небезопасно. **Рекомендация**: передавать прокси явным аргументом в HTTP‑слой. |
| L8 | `attempts` инкрементируется при захвате; финальный статус считается по `attempts+1`. | Корректно (согласовано с фильтром `attempts < max` в выборке). |

## 5. Аудит базы данных

- **Движок**: был `MyISAM` → переведён на `InnoDB` (в `addon.xml` для новых установок; ALTER для существующих — см. §2). InnoDB даёт транзакции и строковые блокировки, что важно для очереди и устранения гонок.
- **Внешние ключи отсутствуют**. Целостность поддерживается «уборкой сирот» (`cleanup('orphans')`). **Рекомендация**: при InnoDB добавить FK с `ON DELETE CASCADE` на `product_id`/`user_id`/`category_id` **или** повесить уборку на события удаления товара/категории/пользователя (хуки ядра), а не только на ручную чистку.
- **Индексы** очереди (`status`, `run_after`, `product_user_platform`) корректны для выборки крона. При больших объёмах рассмотреть составной индекс `(status, run_after)`.
- **Типы**: `payload mediumtext` достаточно для снимка карточки. `error text` — ок.
- **Чистка**: `TRUNCATE`/`DELETE` присутствуют; для очень больших таблиц `DELETE` батчами во избежание долгих блокировок.

## 6. Ограничения внешних API и их обработка

Ниже — типовые ограничения и как с ними обходиться. **Конкретные числа проверяйте в актуальной документации площадок.**

- **ВКонтакте** (`wall.post` + загрузка фото): нужен токен с правами на стену сообщества (`wall`, `photos`), для публикации от имени группы — `from_group=1` и корректный `owner_id` со знаком «минус». Лимиты частоты на метод. Обработка: бэк‑офф при `error_code` (особенно `6` Too many requests), уважать `captcha_*`/`error_code 14`. Сейчас фото грузится по схеме getWallUploadServer→upload→saveWallPhoto — корректно.
- **Instagram (Graph API)**: требуется **бизнес/Creator‑аккаунт**, привязанный к Facebook‑странице, разрешение на content‑publishing, и **публичный `image_url`** (Instagram скачивает картинку сам — локальные/приватные URL не подойдут). Есть **суточный лимит публикаций** на аккаунт и асинхронная готовность контейнера. Обработка: проверять статус контейнера перед `media_publish`, считать суточные публикации, понятные ошибки прав/ревью.
- **Telegram (Bot API)**: подпись фото ограничена **1024 символами** (учтено), общий текст — 4096. Есть **частотные лимиты** (примерно ~1 сообщение/сек в чат, всплески ограничены; рассылки — отдельные лимиты). Бот должен быть админом канала. Обработка: уважать `429` и поле `retry_after`.
- **Одноклассники** (`mediatopic.post`): запросы подписываются MD5 (реализовано). **Загрузка фото не реализована** — нужен поток `photosV2.getUploadUrl` → upload → `photos.commit`, затем вставка `photo`‑медиа в attachment. Обработка ошибок по `error_code`.
- **Pinterest (API v5)**: Bearer‑токен, нужен валидный `board_id`, **rate limits** на приложение/пользователя, по `media_source=image_url` картинка должна быть доступна. Обработка: `429` + бэк‑офф.
- **MAX (webhook)**: формат полезной нагрузки зависит от настройки приёмника; успех определяется HTTP‑кодом. Обработка: ретраи на 5xx, журналирование тела ответа.
- **YouTube (Data API v3)**: расход **квоты на загрузку значителен** (одна загрузка «съедает» большую часть суточной квоты проекта), плюс суточный лимит проекта. Нужны OAuth `client_id/secret` и **refresh_token** (access‑token обновляется — реализовано), resumable‑upload (реализовано). Учитывайте требования к проверкам и статусам приватности. Обработка: считать квоту, не ретраить «quotaExceeded» немедленно (бэк‑офф до суток), хранить `videoId`.
- **RuTube (Partner API)**: эндпоинт/доступ партнёрские и могут отличаться от использованного `https://rutube.ru/api/video/` — **уточняйте у RuTube**. Обработка: трактовать как потенциально нестабильный, журналировать ответы.

### Сквозные рекомендации по публикации через API

1. **Идемпотентность**: хранить внешний id ответа (post_id/media id/video id) в задаче, чтобы повтор не плодил дубли (для платформ с поддержкой idempotency key — использовать его).
2. **Бэк‑офф уже добавлен**; дополнительно — соблюдать `Retry‑After`/`retry_after` из ответов (429/503).
3. **Rate limiting на стороне модуля**: пер‑площадочный «не чаще N/мин» (например, через `run_after`‑планирование), чтобы не упираться в лимиты.
4. **Dead‑letter**: задачи, исчерпавшие попытки, уже получают статус `error`; полезна отдельная вкладка/уведомление и кнопка «повторить».
5. **Раздельные прокси по площадкам** (некоторые сервисы чувствительны к региону/репутации IP).
6. **Наблюдаемость**: записи уже идут в журнал событий ядра; полезно агрегировать ошибки по кодам.
7. **Секреты**: токены теперь шифруются; не логировать их тела (сейчас логируются метаданные/ответы — проверьте, что в ответах площадок не печатаются секреты).

## 7. Рекомендации по дальнейшему развитию

1. Перенести HTTP‑слой на инъекцию прокси аргументом (убрать `static`), добавить пер‑площадочные таймауты и лимит размера загрузок.
2. Реализовать загрузку фото для OK (`photosV2`).
3. Заменить `REPLACE INTO` на `INSERT ... ON DUPLICATE KEY UPDATE`.
4. Добавить FK/каскады при InnoDB либо уборку сирот по хукам удаления.
5. Анти‑SSRF для `image_url`/`video_url` (блокировка приватных диапазонов, запрет приватных редиректов).
6. Подключить HTMLPurifier для «подробного описания» (точнее текущего санитайзера).
7. Вынести cron‑пароль из URL (заголовок/CLI), `hash_equals` для `feed_secret`.
8. Вкладка «повторить ошибочные» и пер‑площадочные счётчики лимитов в аналитике.

## 8. Проверка качества внесённых правок

- Все PHP‑файлы прошли проверку сбалансированности скобок (в песочнице нет PHP‑бинарника и сети, поэтому `php -l` недоступен — выполнена статическая проверка структуры; перед продакшеном прогоните `php -l` и установку на dev‑копии).
